# YABILABB

**Yet Another BILA But Better** - a lightweight replacement for Bizkaia's BILA tax declaration platform.

Generates `.349` submission files for Hacienda Foral de Bizkaia without requiring Java, BILA, or any heavy GUI. Works natively on macOS, Linux, and Windows.

> **Disclaimer:** This is not official tax software. It is not affiliated with, endorsed by, or supported by the Diputacion Foral de Bizkaia or any government agency. The generated files may contain errors. Always verify your declarations before submission. Use at your own risk.

## Why

BILA is the official Java 1.8 application from the Diputacion Foral de Bizkaia for preparing tax declarations (modelos). It's bloated, slow, barely runs on Apple Silicon Macs, and downloads updates on every launch. All you really need is to produce the submission files.

YABILABB generates those files directly from simple YAML definitions or via a lightweight web UI.

## Supported Modelos

| Modelo | Description | Status |
|--------|------------|--------|
| 349 | Declaracion recapitulativa de operaciones intracomunitarias | Done |

The architecture supports adding more modelos. See [Adding a new modelo](#adding-a-new-modelo).

## Installation

Requires Python 3.12+.

```bash
# Clone and install
git clone <repo-url> && cd YABILABB
uv venv && uv pip install -e ".[web,dev]"

# Or without uv
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[web,dev]"
```

## Usage

### CLI

```bash
# Generate a .349 file from a YAML declaration
yabilabb generate declaration.yaml

# Generate with explicit output path
yabilabb generate declaration.yaml -o 2025T02_12345678.349

# Parse an existing .349 file (from BILA) to YAML
yabilabb parse existing_file.349 -o parsed.yaml

# List supported modelos
yabilabb modelos
```

### Web UI

```bash
yabilabb serve
# Open http://127.0.0.1:8000
```

The web UI lets you:
- Select a modelo from the home page
- Create new declarations by filling in declarant info
- Add/remove intra-community operators with live updates (htmx)
- Import existing `.349` files generated by BILA
- Generate and download `.349` files for submission

Declarations are saved as YAML files under `~/.yabilabb/declarations/<modelo>/`.

### YAML Format

```yaml
exercise_year: 2026
period: "1T"    # 1T-4T quarterly, 01-12 monthly

declarant:
  nif: "B12345678"
  name: "EMPRESA EJEMPLO SL"
  phone: "944000000"
  contact_name: "GARCIA LOPEZ JUAN"
  email: "juan@ejemplo.com"

operators:
  - country: DE
    nif: "279463621"
    name: "GERMAN COMPANY GMBH"
    key: A         # See operation keys below
    amount: 4234.71

  - country: FR
    nif: "12345678901"
    name: "SOCIETE FRANCAISE SARL"
    key: S
    amount: 15000.00
```

### Operation Keys (Modelo 349)

| Key | Description |
|-----|-------------|
| E | Entregas intracomunitarias exentas (exempt deliveries) |
| A | Adquisiciones intracomunitarias sujetas (acquisitions) |
| S | Prestaciones de servicios (services provided) |
| I | Adquisiciones de servicios (services acquired) |
| T | Operaciones triangulares (triangular operations) |
| M | Entregas post-importacion exenta |
| H | Like M, via fiscal representative |
| R | Transferencias bienes en consigna |
| D | Devoluciones bienes en consigna |
| C | Sustituciones empresario/profesional destinatario |

## Docker

```bash
# Build
docker build -t yabilabb .

# Run with persistent storage
docker run -p 8000:8000 -v yabilabb-data:/data yabilabb
```

Open http://localhost:8000. Declarations are stored in the `yabilabb-data` volume and persist across container restarts.

To generate files via CLI inside the container:

```bash
docker run -v $(pwd):/work -v yabilabb-data:/data yabilabb \
  yabilabb generate /work/declaration.yaml -o /work/output.349
```

## File Format

A `.349` file is a ZIP archive containing one `.tmp` file with:

1. **XML envelope** (`<ENVIO>`) with metadata, form field values, and an MD5 hash
2. **Fixed-width 500-character records** (inside `<DATOS>/<SUBREG>` tags) per the [BOB 2020 Orden Foral 570/2020](https://www.bizkaia.eus/lehendakaritza/Bao_bob/2020/03/11/I-197_cas.pdf) specification (Anexo II, pages 14-27)

Record types:
- **Type 1** (1 per file): Declarant header with summary totals
- **Type 2** (N per file): One per intra-community operator + operation key
- **Type 2 Rectifications**: Corrections to previously declared amounts

The generated files are byte-compatible with BILA output in the DATOS section (verified by tests against real BILA-generated examples).

## Testing

```bash
pytest
```

Tests include byte-for-byte comparison of generated records against a real BILA-generated example file. Place your own BILA examples under `examples/` (gitignored) for roundtrip testing.

## Project Structure

```
src/yabilabb/
  models.py      Pydantic data models (Declaration, Operator, Rectification)
  records.py     Fixed-width 500-char record generation per BOB 2020 spec
  envelope.py    BILA XML envelope builder (RC, R0, RD, DATOS sections)
  writer.py      ZIP packager - produces .349 files
  parser.py      Parses existing .349 files back to models
  yaml_io.py     YAML serialization/deserialization
  cli.py         CLI entry point with modelo registry
  web/
    app.py       FastAPI application with modelo-scoped routes
    templates/   Jinja2 templates (htmx for interactivity)
    static/      CSS
```

## Adding a New Modelo

1. Add the modelo to the `MODELOS` registry in `cli.py`
2. Create modelo-specific modules (e.g., `records_303.py`, `models_303.py`) following the same pattern as the 349 implementation
3. Add a writer function and parser function
4. Wire them into the CLI and web UI

The web UI routes are already scoped by modelo (`/modelo/{codigo}/declarations/...`), and declarations are stored in separate directories per modelo.
